# ADR-004: Cost Optimization Strategy — Prompt Caching, Haiku Routing, Batch API

**Date :** 2026-03-04
**Status :** Accepted
**Deciders :** Emmanuel Smadja (Fondateur)

---

## Contexte

Freenzy.io execute 24 agents IA (12 business + 12 personal + 4 L2 + 4 L3) avec des appels frequents a l'API Anthropic. Avec la montee en charge de la plateforme, les couts LLM deviennent significatifs. Trois optimisations complementaires ont ete identifiees pour reduire les couts sans degrader la qualite.

---

## Decisions

### 1. Prompt Caching Anthropic (−90% sur tokens systeme)

**Decision :** Activer le prompt caching Anthropic sur tous les system prompts des agents.

**Mise en oeuvre :**
- Header `anthropic-beta: prompt-caching-2024-07-31` sur le client Anthropic
- System prompt converti en `TextBlockParam[]` avec `cache_control: { type: 'ephemeral' }`
- Fichier : `src/core/llm/llm-client.ts`

**Mecanisme :** Anthropic cache les blocs marques `ephemeral` pendant 5 minutes. Les appels subsequents paient 0.1x le prix d'entree normal sur les tokens caches.

**Impact :** Pour un agent avec system prompt de 500 tokens appele 100x/jour :
- Sans cache : 50 000 tokens d'entree
- Avec cache : 500 (ecriture) + 99 × 50 (lecture cache) = 5 450 tokens equivalents
- **Economie : ~89%** sur les tokens systeme

**Suivi :** `cacheReadTokens` et `cacheCreatedTokens` loggues dans chaque reponse LLM.

---

### 2. Routing de modele : Tier ultra-fast (Haiku) pour taches simples

**Decision :** Ajouter un tier `ultra-fast` mappant vers `claude-haiku-4-5-20251001` pour les taches de classification et reponses courtes.

**Comparaison des couts :**

| Modele | Input (credits/1M) | Output (credits/1M) | Ratio vs Haiku |
|--------|-------------------|---------------------|----------------|
| Haiku | 80 | 400 | 1x |
| Sonnet | 300 | 1 500 | ~3.75x |
| Opus | 1 500 | 7 500 | ~18.75x |

**Taches migrees vers Haiku :**
- `fz-repondeur` classification de messages entrants (type, urgence, categorie) — JSON ≤ 512 tokens
- `fz-repondeur` reponses simples (priorite low + classification general + non urgent) — ≤ 512 tokens
- `fz-budget` categorisation de transactions (assign categorie + confiance) — ≤ 1024 tokens

**Taches conservees sur Sonnet :** reponses complexes repondeur, analyses budget, projections, rapports L2.
**Taches conservees sur Opus :** L3 executif + extended thinking.

**Fichiers modifies :**
- `src/core/llm/llm.types.ts` — `ModelTier` etendu
- `src/core/llm/llm-client.ts` — `modelMap` mis a jour
- `src/billing/pricing.ts` — prix Haiku ajoutes
- `src/agents/level1-execution/repondeur/repondeur.agent.ts`
- `src/agents/level1-execution/personal/budget/budget.agent.ts`

**Prompts dedies :**
- `REPONDEUR_HAIKU_SYSTEM_PROMPT` — -55% vs prompt Sonnet (regles essentielles uniquement)
- `BUDGET_HAIKU_CATEGORIZE_SYSTEM_PROMPT` — -50% vs prompt Sonnet complet

---

### 3. Batch API Anthropic (−50% sur traitements asynchrones)

**Decision :** Utiliser l'API Batch Anthropic pour les summaries du repondeur quand 3+ utilisateurs sont actifs simultanement.

**Mecanisme :**
- Reduction de 50% sur input + output tokens
- Traitement asynchrone (resultats en 5-60 min, max 24h)
- Resultat poll par le job cron `check_batch_results` (toutes les 15 min)

**Conditions d'activation :**
- Minimum 3 configs actives dans le meme batch cycle
- Redis disponible (stockage des metadata batch)
- Sinon : fallback vers event bus (immediat, prix normal)

**Tracking :**
- Batches actifs en Redis : cle `llm:active_batches` → liste de batchIds
- Metadata par batch : `llm:batch:repondeur:{batchId}` → JSON (TTL 26h)
- `customId` encode : `repondeur-{frequency}-{configId}-{userId}-{timestamp}`

**Nouveau fichier :** `src/core/llm/llm-batch.ts`
**Cron job ajoute :** `check_batch_results` (toutes les 15 min)

---

### 4. Redis Memoization des reponses LLM (deduplication)

**Decision :** Cacher les reponses LLM identiques dans Redis pour eviter les appels dupliques.

**Activation :** Optionnelle via `enableMemoization: true` dans `LLMRequest`.
**Cle :** `llm:memo:{sha256(model + systemPrompt + userMessage)[:32]}`
**TTL :** Configurable via `memoizationTtlSeconds` (defaut: 300s = 5 min)

**Cas d'usage :**
- Meme message WhatsApp recu plusieurs fois (resubmit)
- Meme relevé bancaire soumis deux fois a budget agent
- Status checks identiques du monitoring

**Fichier :** `src/core/llm/llm-client.ts` — `buildMemoKey()` + lecture/ecriture Redis avant/apres appel API.

---

## Consequences

**Positives :**
- Reduction des couts LLM significative (economie estimee 40-70% selon volume)
- Architecture transparente : agents n'ont pas besoin de savoir quel modele est utilise
- Haiku prompts plus courts = meilleure fiabilite sur taches simples
- Batch API dechaine la puissance du repondeur a grande echelle

**Negatives / Risques :**
- Haiku peut produire des resultats moins nuances sur certaines classifications edge cases
- Le TTL de 5 min du prompt cache implique un premier appel plus couteux apres chaque expiration
- Batch API introduit une latence (5-60 min vs temps reel) pour les summaries batch

**Mitigation :**
- Fallback gracieux : si Haiku echoue a parser JSON → fallback valeur par defaut (deja prevu dans le code agent)
- Redis unavailable → memoization silencieusement desactivee (try/catch)
- Batch unavailable → fallback event bus immediat

---

## Alternatives rejetees

- **Mistral/Llama** : Complexite de maintenance multi-provider, qualite inferieure sur le francais
- **Caching complet de contexte** (conversations longues) : Complexite trop elevee pour le gain actuel
- **Compression de prompts aggressive** : Risque de degrader la qualite des reponses agents
